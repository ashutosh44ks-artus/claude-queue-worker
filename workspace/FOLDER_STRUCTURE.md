# Folder Structure

This document describes the folder structure conventions to follow when developing this application.

---

## Overview

```
/
├── app/
├── components/
│   └── ui/
├── lib/
│   ├── stores/
│   └── supabase/
├── types/
└── public/
```

---

## `app/`

Next.js App Router directory. All routes, layouts, and server actions live here.

```
app/
├── layout.tsx          # Root layout — fonts, global providers
├── page.tsx            # Home page (/)
├── globals.css         # Global Tailwind CSS styles
├── favicon.ico
└── [route]/
    ├── page.tsx        # Route UI (Server Component by default)
    └── actions.ts      # Server Actions for this route
```

**Conventions:**
- Each route gets its own folder under `app/`.
- Server Actions for a route go in `actions.ts` alongside its `page.tsx`.
- Prefer Server Components. Add `"use client"` only when necessary (interactivity, hooks, browser APIs).
- Do not fetch data with `useEffect` — use Server Components or TanStack Query.

---

## `components/`

Shared React components used across multiple routes.

```
components/
├── providers.tsx       # Global context providers (QueryClient, Sonner, etc.)
└── ui/                 # shadcn/ui primitive components
    ├── button.tsx
    ├── sonner.tsx
    └── ...
```

**Conventions:**
- `ui/` contains only shadcn/ui primitives. Do not put business logic here.
- Feature-specific components that are reused across routes go directly in `components/`.
- If a component is only used within a single route, co-locate it inside that route's folder under `app/`.
- Use PascalCase for all component file names and exports.

---

## `lib/`

Shared utilities, configurations, and non-UI modules.

```
lib/
├── utils.ts            # cn() helper for merging Tailwind classes
├── stores/
│   └── counter-store.ts  # Zustand stores for global UI state
└── supabase/
    ├── client.ts       # Supabase browser client (use in Client Components)
    └── server.ts       # Supabase server client (use in Server Components / Actions)
```

**Conventions:**
- `lib/utils.ts` is for small, general-purpose utility functions.
- `lib/stores/` contains Zustand stores. One file per domain/feature.
- Always use `@/lib/supabase/server` in Server Components and Server Actions.
- Always use `@/lib/supabase/client` in Client Components.
- Do not mix server and client Supabase imports.

---

## `types/`

Global TypeScript type definitions.

```
types/
├── database.types.ts   # Auto-generated Supabase DB types (do not edit manually)
└── env.d.ts            # Environment variable type declarations
```

**Conventions:**
- `database.types.ts` is generated by the Supabase CLI — do not edit it manually.
- Add new environment variables to `env.d.ts` to keep them typed.
- Route- or feature-specific types should be co-located with the code that uses them, not placed here.

---

## `public/`

Static assets served at the root URL.

```
public/
├── next.svg
├── vercel.svg
└── ...
```

**Conventions:**
- Images, fonts, and other static files go here.
- Reference them with a leading `/` (e.g., `/logo.png`), not relative paths.

---

## General Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Components | PascalCase | `ProductCard.tsx` |
| Hooks | camelCase with `use` prefix | `useProducts.ts` |
| Utilities / helpers | camelCase | `formatDate.ts` |
| Zustand stores | camelCase with `-store` suffix | `cart-store.ts` |
| Route folders | kebab-case | `app/product-detail/` |
| Server Action files | `actions.ts` | `app/checkout/actions.ts` |

---

## State Management Guidelines

| State type | Tool |
|---|---|
| Server / async data | TanStack Query |
| Form state | React Hook Form + Zod |
| Global UI state | Zustand |
| Local component state | `useState` / `useReducer` |

---

## Adding a New Feature

1. Create a route folder: `app/[feature-name]/`
2. Add `page.tsx` (Server Component) for the UI.
3. Add `actions.ts` for any mutations (Server Actions).
4. Add reusable components to `components/` if shared, or co-locate them in the route folder.
5. Add a Zustand store in `lib/stores/` only if global state is needed.
6. Add any new Zod schemas alongside the code that uses them.
